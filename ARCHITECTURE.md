# 아키텍처 및 의사결정 기록

## 목표

"사전학습/공개 모델을 최대한 활용해 SOTA에 가까운 방식으로, 가입/API 키가 필요 없는 무료 자원만으로 SK하이닉스 주가를 예측하고 GitHub Pages로 배포한다"는 요구사항에 따라 설계했다.

## 파이프라인

```
yfinance (가격) ─┐
                  ├─► forecast.py (Chronos 예측 + 감성 보정) ─► report.py (정적 HTML) ─► docs/
Google News RSS ─┴─► sentiment.py (뉴스 감성 점수)
```

## 모델 선택과 이 실행 환경의 제약

이 프로젝트를 개발한 실행 환경(샌드박스)은 **프로세스당 메모리 할당 한도가 약 500~700MB** 수준으로 매우 제한적이었다 (호스트 전체 메모리는 30GB 이상 여유가 있었지만, 개별 프로세스는 그보다 훨씬 낮은 한도에 걸림 — `numpy` 배열을 500MB~800MB 사이 크기로 할당하면 실패하는 것으로 직접 확인함).

이 제약이 모델 선택에 실제로 영향을 줬다:

- **감성분석**: 원래 금융 특화 한국어 모델인 `snunlp/KR-FinBert-SC`(BERT-base, ~110M 파라미터, fp32 기준 약 440MB)를 쓰려 했으나, 모델 가중치 로딩 직후 메모리 할당이 간헐적으로 실패했다(스레드 수를 1로 제한해도 동일). 대신 `monologg/koelectra-small-finetuned-nsmc`(ELECTRA-small, ~14M 파라미터)로 교체해 안정성을 확보했다. **트레이드오프**: 이 모델은 영화 리뷰(NSMC) 데이터로 학습된 범용 감성분류기라 금융 뉴스 도메인에는 최적화돼 있지 않다. 실제 서비스로 발전시킨다면 더 여유로운 환경에서 금융 특화 모델(FinBERT류)로 교체하는 것을 권장한다.
- **가격 예측**: `amazon/chronos-t5-tiny`(8M)와 `amazon/chronos-t5-small`(46M) 둘 다 이 환경에서 안정적으로 로딩/추론이 확인되어, 정확도가 더 나은 **small**을 채택했다. 학습이 필요 없는 zero-shot 파운데이션 모델이라는 점에서 "사전학습 모델을 최대한 활용"이라는 목표에 부합한다.

## 뉴스 감성 → 예측 보정 방식

Chronos는 단변량(가격 시계열)만 입력받는 파운데이션 모델이라 뉴스 감성을 직접 학습에 결합할 수 없다. 대신 `forecast.py`에서 Chronos의 예측 중앙값에 뉴스 평균 감성 점수(-1~1)를 곱셈 보정으로 반영한다:

```
adjustment = 1.0 + sentiment_score * 0.01   # 감성 점수 1.0당 최대 ±1%
```

이는 **학습된 결합 모델이 아니라 단순 휴리스틱**이다. 감성이 강하게 긍정/부정적일수록 예측을 같은 방향으로 소폭 밀어주는 수준이며, 정교한 멀티모달 예측을 대체하지 않는다. 추후 개선한다면 가격+감성을 함께 넣어 학습하는 모델(예: 감성 점수를 covariate로 받는 시계열 모델)로 바꾸는 것이 자연스러운 다음 단계다.

## 성능 검증 (백테스트)

예측이 실제로 쓸만한지 확인하기 위해 `src/backtest.py`에 두 가지 검증을 둔다. 둘 다 **예측 시점에는 그 이후의 실제 값을 전혀 사용하지 않는다** (look-ahead 없음) — 미래 정보가 새어 들어가면 검증 결과가 의미 없어지기 때문에 이 원칙을 지키는 것이 핵심이다.

- `backtest()`: 가장 최근 5거래일을 정답으로 숨기고, 그 이전 데이터까지만으로 한 번 예측해 MAE/MAPE를 낸다. 직관적이지만 표본이 하나뿐이라 우연에 좌우될 수 있다.
- `rolling_backtest()`: 과거 시점 10곳을 5일 간격으로 순회하며 각각 "그 시점까지의 데이터만" 사용해 반복 예측/채점한다(walk-forward). 평균과 표준편차를 함께 보고해 단일 구간보다 신뢰할 수 있는 성능 추정치를 준다. 실제 결과는 평균 MAPE 약 10~11% 수준으로, 방향성(상승/하락)은 어느 정도 맞추지만 변동성이 큰 구간에서는 오차가 커진다.

## 배포 방식

GitHub Pages는 정적 파일만 서빙할 수 있으므로, Python 파이프라인은 로컬(혹은 CI)에서 실행해 `docs/index.html`과 `docs/assets/chart.png`를 생성한 뒤 그 결과물을 커밋해서 push하는 방식을 택했다. 별도 서버나 GitHub Actions 없이 가장 단순하게 동작한다.
