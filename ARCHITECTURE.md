# 아키텍처 및 의사결정 기록

## 목표

공개된 사전학습/통계/머신러닝 모델과 시장 데이터를 결합해 SK하이닉스 주가를 다각도로
분석하고, GitHub Pages로 정적 리포트를 배포한다. 가입/API 키가 필요 없는 무료 자원만 쓴다.

## 파이프라인

```
yfinance (SK하이닉스 가격) ──┬─► forecast.py (Chronos+추세모델 앙상블, 기술적지표 GBM 비교) ─┐
yfinance (시장지수/섹터지수/동종업계) ─► related_stocks.py, index_scan.py (시장맥락 진단) ──┤
Google News RSS (국문 7개+영문 7개 검색어) ─► fetch_news.py (중복 통합) ─► sentiment.py ────┼─► report.py ─► docs/
                                                                                         ┘
```

실행은 `stage_collect.py`(데이터 수집·감성분석) → `stage_forecast.py`(가격 예측·백테스트) →
`report.py`(리포트 생성) 순서이며, 앞의 두 단계는 서로 다른 프로세스로 실행된다 (아래
"프로세스 분리" 항목 참고).

## 모델 선택

- **가격 예측(3개 모델 비교, 2개 앙상블)**
  - `amazon/chronos-bolt-base`: Amazon이 공개한 사전학습 시계열 파운데이션 모델. 분위수를
    직접 예측(direct multi-step)해 원조 `chronos-t5`보다 정확하고 빠르다. 이 저장소를 처음
    만든 샌드박스는 프로세스당 메모리 한도가 500~700MB라 더 작은 `chronos-bolt-small`을
    썼으나, 이 환경은 그 제약이 없음(로딩 테스트로 확인, ~19GB 여유)을 확인해 더 큰 `base`로
    올렸다.
  - 감쇠추세 지수평활(`statsmodels.tsa.holtwinters.ExponentialSmoothing`, damped trend):
    Chronos와 독립적인 가정(추세 완만화)에 기반한 고전 통계 모델. 서로 다른 가정을 가진
    모델을 앙상블하면 한쪽이 특정 구간에서 잘못 짚어도 전체 오차 분산이 줄어든다는 것이
    시계열 예측 대회(M4/M5 등)에서 반복 확인된 경험칙이다. `forecast.py`는 이 둘을
    실측 백테스트로 검증된 비중(Chronos 70% / 추세모델 30%)으로 블렌딩한다.
  - 기술적 지표 기반 gradient boosting 분위수 회귀(`technical_model.py`,
    `sklearn.HistGradientBoostingRegressor`): RSI/MACD/이동평균 이격도/변동성 등 표준
    기술적 지표를 입력으로 한다. 퀀트 업계와 시계열 경진대회(M5 등)에서 널리 검증된 방식이며,
    Chronos·추세모델과 완전히 다른 각도(모멘텀/과매수·과매도 레짐)에서 접근하므로 앙상블에는
    넣지 않고 세 번째 비교축으로 리포트에 함께 노출한다.

  `backtest.py`의 walk-forward 롤링 백테스트(12개 구간)로 세 가지를 비교한 결과: 앙상블
  평균 MAPE 약 8~9%, Chronos 단독 약 9~9.5%, naive(직전값 유지) 기준 약 8.7~8.9%. 앙상블이
  Chronos 단독보다는 꾸준히 낫지만, naive와는 오차 크기 기준으로 큰 차이가 없다 — 이는
  단기 금융 시계열이 랜덤워크에 가깝다는, 계량금융에서 잘 알려진 현상과 일치한다. 대신
  방향 적중률(다음 거래일 상승/하락을 맞추는 비율)은 약 60% 안팎으로 naive(방향 예측 자체가
  없음)보다 실전에서 더 쓸모가 있다.

- **뉴스 감성분석(언어별 금융 특화 모델)**
  - 한국어: `snunlp/KR-FinBert-SC` (금융 도메인 특화 BERT-base, ~110M). 과거엔 개발
    샌드박스의 프로세스당 메모리 한도(~500-700MB) 때문에 로딩이 실패해 범용 모델
    (koelectra-small, 영화 리뷰 학습)로 대체했었으나, 이 환경은 그 제약이 없음을 확인해
    다시 금융 특화 모델로 교체했다.
  - 영어: `ProsusAI/finbert`. 금융 뉴스/문장 감성 분류로 가장 널리 쓰이는 사전학습 모델.

## 뉴스 수집 범위와 중복 처리

`fetch_news.py`는 한국어 7개(`SK하이닉스`, `SK하이닉스 목표주가`, `SK하이닉스 실적`,
`HBM 고대역폭메모리`, `반도체 지정학`, `반도체 업황`, `메모리 반도체 가격`) + 영어 7개
(`SK Hynix`, `SK Hynix price target analyst`, `SK Hynix earnings`, `HBM AI chip demand`,
`memory chip export controls China`, `semiconductor memory chip industry outlook`,
`DRAM NAND price`) 검색어로 뉴스를 모은다. 영어 검색은 Google News 미국판(en-US)을 쓰는데,
Reuters/Bloomberg/CNBC뿐 아니라 Nikkei Asia(일본), SCMP(중국) 등 영문으로 발행되는 아시아
매체도 함께 잡혀 별도 언어별 모델 없이 미/일/중 시각을 폭넓게 커버한다.

검색어 14개를 각각 최대 30건씩 가져오면(최대 420건) 같은 사건을 여러 매체가 받아쓴 기사가
크게 중복된다. `_deduplicate()`는 언어별로 제목(매체명 접미사 제거 후) 유사도가
75% 이상이면 하나로 합치고, 여러 검색어에 걸쳐 잡힌 경우 더 높은 검색어 가중치를 취한다.
이렇게 하지 않으면 화제성이 큰 사건 하나가 감성 평균을 지배해 버린다.

## 뉴스 감성 → 예측 보정 방식

`forecast.py`의 앙상블은 가격 시계열만 입력받으므로, 뉴스 감성 점수를 예측 중앙값에
곱셈 형태로 보정한다. 이 보정은 세 겹의 가중치로 계산된다(`sentiment.py`의
`weighted_average_sentiment`):
1. 분류기 확신도 (예: 98% 확신 긍정 vs 51% 확신 긍정)
2. 검색어 중요도 — 목표주가/실적 뉴스는 일반 산업 뉴스보다 주가에 더 직접적인 신호
3. 최신성 — 게재일 기준 지수감쇠(half-life 3일)

그리고 예측 horizon이 멀어질수록(거래일이 뒤로 갈수록) 보정 폭 자체도 지수적으로 감쇠시킨다
(`SENTIMENT_DECAY`). 뉴스의 가격 영향력은 며칠 안에 빠르게 옅어진다고 보는 것이 합리적이기
때문이다.

**왜 학습 기반 융합(뉴스를 covariate로 넣어 함께 학습)이 아니라 이 보정 방식을 쓰는가**:
업계에서 검증된 방법은 기본 예측(여기서는 Chronos) 위에 별도 회귀모델이 뉴스/기술적 지표
같은 공변량으로 잔차를 설명하도록 학습시키는 "베이스 예측 + 공변량 회귀" 구조다(AWS
AutoGluon-TimeSeries가 Chronos-Bolt에 대해 공식적으로 지원하는 `covariate_regressor` 패턴이
그 예다). 기술적 지표는 전체 가격 이력에서 바로 계산할 수 있어 이 구조를 그대로
`technical_model.py`에 적용했다. 반면 뉴스 감성을 이 구조로 학습시키려면 "과거 각 시점에
실제로 존재했던 뉴스"의 아카이브가 필요한데, Google News RSS는 실시간 검색 결과만 제공하고
과거 특정일 기준 스냅샷을 대량으로 재현하려면 시점마다 별도 조회가 필요해(수백~수천 회
호출) 무료·비로그인 환경에서는 현실적이지 않다. 그래서 뉴스는 실시간 시점에서 계산한
가중평균 감성 점수를 예측치에 곱하는 보정(overlay) 방식으로 반영한다. 이는 RavenPack류
뉴스 감성 지수를 쓰는 실무 퀀트 전략에서도 흔한 패턴(별도로 계산한 감성 점수를 기존
정량 예측에 오버레이)과 같은 결이다.

## 시장 맥락: 시장 전체 vs 섹터 vs 개별 이슈

`related_stocks.py`는 SK하이닉스의 움직임이 어느 층위에서 비롯됐는지 구분하기 위해
비교군을 세 계층으로 나눈다.
- 시장 전체: 코스피(`^KS11`), S&P500(`^GSPC`), 나스닥(`^IXIC`)
- 반도체 섹터: 필라델피아반도체지수(`^SOX`)
- 메모리 동종업계: 삼성전자, Micron, SanDisk

`diagnose_move()`는 최근 5거래일 누적수익률을 계층별 평균과 비교해, 하이닉스 고유의 움직임이
컸는지(시장·섹터·동종업계 대비 괴리가 큼) 아니면 전체 흐름에 동조했는지 판단한다.

`index_scan.py`는 여기서 한 걸음 더 나가, 코스피200/나스닥100/S&P500 각각의 대표종목
표본(대형주·업종 대표주 20여 개씩)과 SK하이닉스의 상관관계를 계산해 지수별 상위 종목을
보여준다. 지수 전체(수백 종목)를 매번 스캔하지 않는 이유는 실행 시간과 yfinance 호출 부담
때문이며, 표본 안에서의 순위임을 리포트에 명시한다.

## 성능 검증 (백테스트)

`backtest.py`는 예측 시점 이후의 실제 값을 전혀 사용하지 않는(look-ahead 없음) 두 가지
검증을 제공한다.
- `backtest()`: 최근 5거래일을 정답으로 숨기고 그 이전 데이터까지만으로 예측, naive(직전값
  유지) 기준과 방향 적중률을 함께 계산한다.
- `rolling_backtest()`: 과거 12개 시점을 5일 간격으로 순회하며 walk-forward 검증한다.
  앙상블·Chronos 단독·naive 세 가지 MAPE와 방향 적중률을 함께 낸다.

## 프로세스 분리 (main.py)

감성분류기 2종(한국어/영어, 합쳐서 ~900MB)을 로드해 뉴스 감성을 계산한 뒤, **같은 프로세스
안에서 이어서** Chronos-Bolt-base(~800MB)를 로드하면 이 개발 환경(Windows, Python 3.14)에서
간헐적으로 세그멘테이션 폴트 또는 "페이징 파일이 너무 작습니다" 오류가 발생했다. 원인은
PyTorch의 CPU 메모리 할당자가 해제된 텐서를 OS에 곧바로 반환하지 않아, 두 모델의 메모리
풋프린트가 한 프로세스 안에서 누적되기 때문으로 보인다. `concurrent.futures.ProcessPoolExecutor`로
자식 프로세스만 분리해도(부모 프로세스는 계속 살아있으므로) 문제가 재현됐다.

가장 확실히 동작한 해결책은 두 단계를 **완전히 독립된 프로세스**로 나누는 것이다:
`stage_collect.py`(가격/뉴스/감성/시장비교 수집)가 끝까지 실행되고 완전히 종료된 뒤에야
`stage_forecast.py`(Chronos+앙상블 예측·백테스트)가 새 프로세스로 시작된다. `main.py`는
`subprocess.run()`으로 이 둘을 순서대로 실행하고, 중간 결과는 JSON 파일로 주고받는다.
이렇게 하면 앞 단계 프로세스가 종료되며 OS가 그 메모리를 확실히 회수한 뒤 다음 단계가
시작되어 문제가 재현되지 않았다.

## 배포 방식

GitHub Pages는 정적 파일만 서빙하므로, Python 파이프라인은 로컬(혹은 CI)에서 실행해
`docs/index.html`과 `docs/assets/*.png`를 생성한 뒤 그 결과물을 커밋해서 push하는 방식을
택했다. 별도 서버나 GitHub Actions 없이 가장 단순하게 동작한다.
